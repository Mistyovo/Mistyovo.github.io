二进制文件

可执行文件的定义

广义：文件中的数据是可执行代码的文件：.out、.exe、.sh、.py

狭义：文件中的数据是机器码的文件：.out、.exe、.dll、.so

可执行文件的分类

Windows：PE（Portable Executable） Linux：ELF（Executable and Linkable Format）
可执行程序 .exe .out
动态链接库 .dll .so
静态链接库 .lib .a

ELF文件格式

ELF文件的类型包括：可执行文件（.exec）、可重定位文件（.rel）、共享目标文件（.dyn）

在审视一个目标文件时，有两种视角可供选择：

链接视角：通过节（Section）来划分，用来划分功能（In Disk）

运行视角：通过段（Segment ）来划分，用来划分读写执行权限（In Memory）

由Linker完成节到段的转化

下面是一些重要的节：

  1. .got（Global Offset Table）：
    • .got 是全局偏移表，用于动态链接。它包含指向动态分配内存地址的指针。当程序需要访问全局变量或静态变量时，通过 .got 表来获取正确的地址。
  2. .plt（Procedure Linkage Table）：
    • .plt 是过程链接表，用于处理动态函数调用。它包含一系列跳转指令，用于首次调用外部函数时解析函数地址，并将解析后的地址存储在 .got 表中。
  3. .symbol（符号表，通常是 .symtab 或 .dynsym）：
    • .symbol 表保存程序中的符号（如函数名、变量名）及其对应的地址。.symtab 用于调试和符号解析，包含所有符号；.dynsym 用于动态链接，包含需要动态解析的符号。

格式化字符串函数

格式化字符串函数可以接受可变数量的参数，并将第一个参数作为格式化字符串，根据其来解析之后的参数。通俗来说，格式化字符串函数就是将计算机内存中表示的数据转化为我们人类可读的字符串格式。几乎所有的 C/C++ 程序都会利用格式化字符串函数来输出信息，调试程序，或者处理字符串。一般来说，格式化字符串在利用的时候主要分为三个部分

  • 格式化字符串函数
  • 格式化字符串
  • 后续参数，可选

漏洞利用

程序崩溃

通常来说，利用格式化字符串漏洞使得程序崩溃是最为简单的利用方式，因为我们只需要输入若干个 %s 即可

%s%s%s%s%s%s%s%s%s%s%s%s%s%s

这是因为栈上不可能每个值都对应了合法的地址，所以总是会有某个地址可以使得程序崩溃。这一利用，虽然攻击者本身似乎并不能控制程序，但是这样却可以造成程序不可用。比如说，如果远程服务有一个格式化字符串漏洞，那么我们就可以攻击其可用性，使服务崩溃，进而使得用户不能够访问。

泄露内存

利用格式化字符串漏洞，我们还可以获取我们所想要输出的内容。一般会有如下几种操作

  • 泄露栈内存
    • 获取某个变量的值
    • 获取某个变量对应地址的内存
    • 修改栈内存上地址指向数据的值：第一步，泄露出字符串在栈上的地址；第二步，将地址写进字符串；第三步，利用%$定位字符串上的这个地址，并修改其指向的值
  • 泄露任意地址内存
    • 利用 GOT 表得到 libc 函数地址，进而获取 libc，进而获取其它 libc 函数地址
    • 盲打，dump 整个程序，获取有用信息。

Linux保护机制及其漏洞和绕过方法

RELRO

RELRO（ReLocation Read-Only）是一种安全机制，通过将程序的重定位表设置为只读来防止GOT（全局偏移表）覆写攻击，分为Partial RELRO（部分只读）和Full RELRO（完全只读），前者仅保护GOT表的一部分，后者则保护所有关键数据结构

STACK CANARY

Stack Canary是一种栈保护机制，旨在防止缓冲区溢出攻击。其原理是在函数调用时，在栈帧顶部插入一个随机值（canary），并在函数返回时检查该值是否被篡改。若被篡改，则表明栈溢出发生，程序将立即停止执行。这种机制通过在函数入口处插入canary值，并在函数返回时验证其完整性来检测栈溢出。Stack Canary并非理论上绝对安全，仅能有效缓解顺序覆盖的栈溢出攻击。

def leak_canary():
    global canary
    canary = b"\x00"
    while len(canary) < 8:
        for x in range(256):
            try:
                with remote("node5.buuoj.cn", 26857) as io:

NX

NX（No-eXecute）是一种安全机制，旨在防止攻击者在堆栈中执行恶意代码。其基本原理是将数据所在内存页标识为不可执行，当程序溢出成功转入shellcode时，程序会尝试在数据页面上执行指令，此时CPU就会抛出异常，而不是去执行恶意指令。NX的实现由软件和硬件共同完成，硬件层利用CPU的NX位，软件层面需要操作系统支持NX以配置页表

ASLR和PIE

ASLR（Address Space Layout Randomization）和PIE（Position-Independent Executable）是计算机安全领域的重要概念，旨在增强程序的安全性。ASLR通过随机化程序在内存中的地址布局，防止攻击者预测内存地址，从而降低缓冲区溢出攻击的成功率。PIE则允许程序在内存中任意位置执行，不依赖于固定的内存地址，这与ASLR结合使用，进一步增强了程序的安全性。

FORTIFY_SOURCE

FORTIFY_SOURCE 是 GCC 和 Clang 编译器提供的一个编译选项，旨在增强 C 和 C++ 程序的安全性，通过在编译期间插入额外的检查来防止常见的缓冲区溢出和其他类型的内存损坏。它通过定义宏 _FORTIFY_SOURCE 来控制对一些函数的调用进行加固，确保在编译时和运行时检测缓冲区溢出和格式化字符串漏洞。FORTIFY_SOURCE 提供两个级别：-D_FORTIFY_SOURCE=1 进行较弱的检查，-D_FORTIFY_SOURCE=2 进行较强的检查，后者在运行时也会进行检查，以防止缓冲区溢出。
